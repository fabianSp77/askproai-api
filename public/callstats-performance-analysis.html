Critical Performance Issues

1. Full Table Scan on Main Query

Location: Line 45 of CallStatsOverview.php

// Current (PROBLEM)
$query = Call::whereDate('created_at', today());
// SQL: WHERE DATE(created_at) = '2025-11-21'

Impact: The DATE() function prevents index usage, forcing MySQL to scan all 967 rows

EXPLAIN Result:

{
    "type": "ALL",           // Full table scan
    "possible_keys": null,   // No index can be used
    "rows": 967,             // Scans entire table
    "Extra": "Using where"
}

Solution:

// Optimized (USE INDEX)
$query = Call::whereBetween('created_at', [
    today()->startOfDay(),
    today()->endOfDay()
]);

Expected Improvement: 80-90% reduction in query time

2. N+1 Query Pattern for Cost Calculation

Location: Lines 76-85

if ($todayCount > 0) {
    $calls = $query->get();  // Fetches ALL records
    foreach ($calls as $call) {
        $todayCost += $costCalculator->getDisplayCost($call, $user);
        // ... profit calculation
    }
}

Impact:

  • Loads all call records into memory
  • Executes PHP logic for each record
  • Memory usage scales linearly with data

Solution: Pre-calculate costs at write time or use database aggregation

3. Missing Critical Index

Column: has_appointment

Query uses: SUM(CASE WHEN has_appointment = 1 ...)

Index exists: No

Impact: Forces row-by-row evaluation

Solution:

ALTER TABLE calls ADD INDEX idx_has_appointment (has_appointment);
