❌ Was die Middleware NICHT kann

▼

⚠️ Limitierungen & Constraints

  • ❌ Keine Video-Calls: Nur Voice AI über Retell, kein Zoom/Teams Integration
  • ❌ Keine E-Commerce: Stripe Payment nur für Balance Top-Ups, kein Shop
  • ❌ Keine File Uploads: Keine Dokumenten- oder Bild-Uploads via API
  • ❌ Keine Echtzeit-Kalender-UI: Nur API-Integration, kein Frontend-Kalender
  • ❌ Keine automatische Spracherkennung: Deutsche Formate müssen manuell geparsed werden
  • ❌ Keine Cross-Tenant Queries: Super Admin muss Scope explizit bypassen
  • ❌ Keine Bulk Operations: Keine Massen-Buchungen/-Stornierungen
  • ❌ Keine GraphQL: Nur REST API, kein GraphQL Endpoint

🔧 Edge Cases & Bekannte Issues

  • Retell IP Whitelist zu breit: 100.20.0.0/16 CIDR = 65,536 IPs (VULN-006)
  • Anonymous Caller Limitation: Nur Same-Call Appointments können modifiziert werden (30min Window)
  • Cal.com Reschedule UID Change: Reschedule erstellt NEUEN Booking UID
  • Date Parsing Fallback: Bei Parse-Fehler → Default "tomorrow 10:00"
  • Super Admin Bypass: Keine Audit-Logs für Cross-Tenant Zugriffe (VULN-004)
  • Raw SQL Risk: Direkte SQL-Queries bypassen Global Scopes (VULN-005)

🔍 Customer Matching: 5 Strategien

Wie werden Kunden über mehrere Strategien identifiziert?

Priorität Strategie Security Level Beschreibung
1 Call-Linked Customer HIGHEST call->customer_id bereits gesetzt (vorheriger Match)
2 Phone Number Match HIGH Company-scoped, Last 8 Digits, Rate Limited (3/hour)
3 Exact Name Match LOW Nur für anonymous callers, 100% exact match required
4 Same-Call Metadata MEDIUM metadata->retell_call_id match (30min window)
5 Company + Date Fallback LOWEST Last resort: company_id + appointment_date

🔒 Anonymous Caller Security

Problem: Caller mit unterdrückter Rufnummer können nicht per Phone verifiziert werden.

Lösung: Same-Call Policy - nur Appointments aus dem aktuellen Call (letzten 30 Minuten) dürfen modifiziert werden.

if (call->from_number === 'anonymous') {
    // SECURITY: Only allow modification of appointments from THIS call
    $booking = Appointment::where('metadata->retell_call_id', $callId)
        ->where('created_at', '>=', now()->subMinutes(30))
        ->first();

    if (!$booking) {
        return "Für ältere Termine rufen Sie bitte direkt an.";
    }
}

✅ Loop Prevention

Verhindert Sync-Loops zwischen Cal.com und CRM:

// When creating appointment from Retell API
$appointment->sync_origin = 'retell_api';
$appointment->calcom_sync_status = 'pending';

// Push to Cal.com
$response = CalcomService::createBooking(...);
$appointment->calcom_sync_status = 'synced';

// When Cal.com webhook arrives
if ($appointment->sync_origin === 'retell_api') {
    // Do NOT push back to Cal.com (already there!)
    Log::info('Skipping Cal.com sync - origin is retell_api');
}

⚠️ Cal.com Reschedule Gotcha

CRITICAL: Cal.com erstellt bei Reschedule einen NEUEN Booking UID!

  • Alter UID wird ungültig
  • Neuer UID muss in calcom_v2_booking_id gespeichert werden
  • Alter UID in metadata->previous_booking_id tracken
