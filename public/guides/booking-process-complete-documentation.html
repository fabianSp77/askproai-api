🎯 Was ist eine Buchung?

Ein Kunde ruft an → Will einen Termin → System prüft Verfügbarkeit → System bucht bei Cal.com → System speichert in DB → Kunde erhält Bestätigung

2. User Journey - Schritt für Schritt

🗣️ SCHRITT 1: Kunde ruft an

User: "Guten Tag, ich möchte einen Termin buchen"

→ Twilio empfängt Anruf
→ Twilio verbindet mit Retell AI
→ Retell startet Agent

🤖 SCHRITT 2: Agent initialisiert

Agent ruft auf:

1. current_time_berlin() → Erhält: current_date, current_weekday, current_time 2. check_customer(call_id: {{call_id}}) → Erhält: customer_name, customer_exists 3. Agent begrüßt: "Guten Tag [Name]! Wie kann ich helfen?"

📅 SCHRITT 3: Terminwunsch erfassen

User: "Nächsten Montag um 14 Uhr für eine Beratung"

Agent sammelt:

  • Datum: "nächsten Montag" → getCurrentDateTimeInfo()
  • Uhrzeit: "14 Uhr" → 14:00
  • Service: "Beratung"
  • Name: Aus check_customer()

✅ SCHRITT 4: Verfügbarkeit prüfen

Agent ruft auf:

collect_appointment_data( call_id: "call_...", name: "Hansi Hinterseer", datum: "2025-10-13", uhrzeit: "14:00", dienstleistung: "Beratung" ) Backend prüft: 1. Parse Datum/Zeit 2. Find Customer by phone 3. Check for duplicates 4. Query Cal.com for availability 5. Return: available=true/false + alternatives

💬 SCHRITT 5: Kunde bestätigt

Agent: "Montag, den 13. Oktober um 14 Uhr ist frei. Soll ich buchen?"

User: "Ja, bitte!"

→ Agent bereitet Buchung vor
→ collect_appointment mit bestaetigung=true

🔄 SCHRITT 6: System bucht

collect_appointment_data( ... SAME parameters ..., bestaetigung: true // ← Trigger für Buchung ) Backend: 1. ✅ Create Cal.com booking 2. ✅ Create Appointment in DB 3. ✅ Link to Call record 4. ✅ Set metadata (created_by, booking_source) 5. ✅ Return: success + booking_id

📧 SCHRITT 7: Bestätigung

Agent: "Ihr Termin wurde erfolgreich gebucht. Sie erhalten eine Bestätigung."

→ Cal.com sendet Email an Kunde
→ Cal.com sendet Email an Staff
→ System logged alles in DB

7. Fehlerquellen - Alle möglichen Probleme

7.1 Timezone Fehler

❌ Symptom:

User will 14:30 Uhr → System sagt "16:30 ist verfügbar" (FALSCH!)

Root Cause:

Cal.com Slot-Zeit nicht von UTC zu Europe/Berlin konvertiert

✅ Fix:

// ÜBERALL wo $slot['time'] gelesen wird: $slotTime = Carbon::parse($slot['time'])->setTimezone('Europe/Berlin'); // GEFIXT in 11 Locations - siehe git-changes.patch

✅ Test:

$utc = Carbon::parse('2025-10-10T12:30:00.000Z'); $berlin = $utc->setTimezone('Europe/Berlin'); echo $berlin->format('H:i'); // Should be: 14:30

7.2 DateTime Parsing Fehler

❌ Symptom:

User: "Nächste Woche Montag" → Agent: "Es gab ein Problem..."

Root Cause (2 Probleme):

  1. args nicht extrahiert → 422 Validation Error
  2. Pattern "nächste Woche Montag" nicht erkannt → 400 Parse Error

✅ Fix 1: args Extraction

// DateTimeInfoController.php:39-42 $args = $request->input('args', []); $data = !empty($args) ? $args : $request->all(); $validated = validator($data, [...])->validate();

✅ Fix 2: Neues Pattern

// DateTimeInfoController.php:124-132 if (preg_match('/^(nächste|diese)\s+woche\s+(montag|...|sonntag)$/i', ...)) { $modifier = $matches[1]; $weekday = $matches[2]; $date = $this->parser->parseRelativeWeekday($weekday, $modifier); return $this->formatSingleDate($date, "{$modifier} Woche {$weekday}"); }

7.3 Duplicate Detection Fehler

❌ Symptom:

User bucht doppelt, oder System sagt "bereits gebucht" fälschlicherweise

✅ Wie es funktioniert:

// RetellFunctionCallHandler.php:1004-1047 $existingAppointment = Appointment::where('customer_id', $customer->id) ->whereDate('starts_at', $appointmentDate->format('Y-m-d')) ->whereTime('starts_at', $appointmentDate->format('H:i:s')) ->whereIn('status', ['scheduled', 'confirmed', 'booked']) ->first(); if ($existingAppointment) { // Generate weekday in German $weekdayName = $existingAppointment->starts_at->locale('de')->isoFormat('dddd'); return { "status": "duplicate_detected", "message": "Ich sehe in meinem System, dass für Ihre Nummer bereits {heute oder am $weekdayName, den $date} um $time Uhr ein Termin eingetragen ist..." }; }

7.4 Cal.com Sync Fehler

❌ Symptom:

DB hat Termin, aber Cal.com nicht (oder umgekehrt)

Root Cause:

Transaction nicht atomic - Cal.com first, DB second

✅ Fix: DB-First Strategy

// RetellApiController.php:699-787 (Cancellation example) DB::beginTransaction(); try { // 1. Update DB first $appointment->update([...]); // 2. Create AppointmentModification AppointmentModification::create([...]); // 3. Update Call record Call::update([...]); DB::commit(); } catch (\Exception $e) { DB::rollBack(); return error("DB update failed"); } // 4. THEN sync to Cal.com (non-blocking) try { $calcom->cancel($bookingId); } catch (\Exception $e) { Log::warning('Cal.com sync failed, DB consistent'); // Continue - DB ist Source of Truth! }
✅ Vorteil:
- DB ist IMMER konsistent (rollback bei Fehler)
- Cal.com Fehler blockiert nicht
- Kann später manuell nachsynchronisiert werden

❌ Symptom:

✅ Fix: Immer Metadata setzen!

'booked_by_user_id' => null, // ✅ NULL für customer // ... ]); // Bei Reschedule: $appointment->update([ // ... 'previous_starts_at' => $oldStartsAt, // ✅ Audit Trail! 'rescheduled_at' => now(), 'rescheduled_by' => 'customer', 'reschedule_source' => 'retell_api', // ... ]); // Bei Cancel: $appointment->update([ // ... 'cancelled_at' => now(), 'cancelled_by' => 'customer', 'cancellation_source' => 'retell_api', 'cancellation_reason' => $reason, // ... ]);

7.6 Anonymous Caller Security

❌ Risiko:

Fremde Personen könnten Termine stornieren/verschieben!

✅ Schutz implementiert:

// RetellFunctionCallHandler.php:1716, 1889 if ($call && ($call->from_number === 'anonymous' || in_array(strtolower($call->from_number ?? ''), ['anonymous', 'unknown', 'withheld', 'restricted', '']))) { // Cancel/Reschedule BLOCKIERT für Anonymous return $this->createAnonymousCallbackRequest($call, $params, $action); } // Booking ist ERLAUBT für Anonymous (neue Kunden!)
Security Design:
  • ✅ Booking: Anonymous erlaubt (Geschäftsanforderung)
  • 🔒 Cancel: Anonymous blockiert → CallbackRequest
  • 🔒 Reschedule: Anonymous blockiert → CallbackRequest
